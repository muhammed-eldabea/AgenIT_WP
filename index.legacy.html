JUST SHIPPED · MAY 2026

VS Code extension — the AgenIT experience inside your editor

AgenIT now ships as a Visual Studio Code extension (packages/vscode-extension/) that surfaces the full terminal experience inside the editor — V-Model pipeline, Mission Squad, Goal loop, 50+ slash commands, @req CodeLens, and a brand-aligned chat surface — without duplicating any backend logic.

  • Compact panel (Ctrl+Alt+A / ⌘⌥A) — embedded web UI with Chat, V-Model, Squad, Goal, Jobs, Memory, Design.
  • Immersive view (Ctrl+Alt+Shift+A / ⌘⌥⇧A) — full-screen webview reproducing the AgenIT design chrome (brand activity bar, right-rail widgets, status pill).
  • Slash command palette (Ctrl+Alt+/ / ⌘⌥/) — fuzzy-find any of the 50+ commands; arguments are prompted via QuickPick / InputBox.
  • Six sidebar TreeViews — V-Model Pipeline, Mission Squad, Goal, Background Jobs, Memory, and Debug Mode (with pending-gate click-to-release).
  • CodeLens for @req SWR-NNN in C / C++ / H — click to jump to the requirement; coverage badge from traceability.json.
  • Native cppcheck diagnostics on save / open for C/C++ files (silent when cppcheck isn't installed).
  • Live events stream — /ws/events subscription refreshes trees on squad / goal / agent ticks without waiting the 2 s poll.
  • Thin-client architecture — the .vsix ships zero orchestrator runtime code. It spawns agenit serve per workspace and talks HTTP + WebSocket. Fix once in @flow/orchestrator, restart the daemon, and the change is live in the extension — no extension rebuild required.

Docs: docs/vscode-extension.md · extension README · built and released via vscode-v* tags. PR #113.

JUST SHIPPED · MAY 2026

Claude CLI support — multi-backend with automatic failover

AgenIT is no longer single-backend. The Claude CLI subprocess backend (packages/gemini-client/src/claude-cli.ts) plugs the locally-installed claude binary into the same pipeline as Gemini CLI — with its own OAuth credentials (Pro / Max / Team / Enterprise subscriptions) and the same hook contract.

  • agenit init --target claude|gemini — seeds .claude/ assets alongside (or instead of) .gemini/. Both targets share .flow/hooks/*.py.
  • Unified hook layer — every .flow/hooks/*.py script now works for both Gemini and Claude CLI. Slopsquatting, citation guard, MISRA gate, undo snapshots — all of it, both backends.
  • Cross-backend fallback chain — on cost-tier exhaustion or hard error, the router auto-falls-back Gemini → Claude → OpenAI. Configurable per tier.
  • Interactive backend picker — agenit init probes which backends you have installed and lets you pick when stdin is a TTY.
  • /backend slash command — diagnostic table of detected / installed / active backends; /backend models <provider> lists available models; /backend chain <tier> shows the failover chain.
  • Status-bar indicator — the REPL footer shows the active backend and current model so you always know who's answering.

Modules: gemini-client/src/backend.ts (selector) · orchestrator/src/backend-detector.ts (probe) · orchestrator/src/commands/backend.ts (slash). PR #116.

MAY 2026

Debug Mode · Harnesses widget · Widget navigation · /serve

AgenIT is now fully observable and keyboard-driven. Every chat, LLM, tool, and agent call becomes a typed Run in a debuggable tree; risky steps can pause for human review; the whole TUI is navigable with Ctrl+W and the control SPA runs inside the REPL.

  • Debug Mode — /debug on turns every chat / LLM / tool / agent call into a typed Run in a tree. Right-rail widget in the TUI, dedicated tab in the Web UI, pause-before gates for stepping, replayable tool runs.
  • Harnesses widget — one-stop add for Skills / MCP / Agents / Tools. Empty template or scaffolding-agent generated, edited in an in-TUI vim editor, installed to the canonical path with MCP-JSON merge + AGENTS.md de-dup.
  • Widget navigation — Ctrl+W enters a keyboard-driven nav layer over the rail widgets. Each widget can expand into a full-pane interactive view (tasks, hardware probes, agents dashboard, recipes). Legacy Ctrl+T / Ctrl+] / Ctrl+G shortcuts preserved.
  • Agents dashboard — new right-rail widget showing every active agent with live activity (reading X, running Y), per-agent token-cost sparkline, session spend total.
  • Recipes — TOML-defined cross-widget orchestration. Fire actions when a tuiBus event matches. Default recipes log goal failures + notify on agent failures; opt-in escalation recipe for hardware probe failures.
  • /serve slash command — start the full control SPA inside the running REPL (same process → shared run-registry → Debug tab sees your live traces). Replaces the need to launch agenit serve in a second terminal.

Docs: docs/debug-mode.md · docs/harnesses.md · docs/widget-navigation.md.

// 02 Objective

Most AI coding tools start every conversation from zero. You re-explain the project, the conventions, the build commands, the bug you fixed last week. AgenIT is built around the opposite premise — the agent learns once and remembers, so each session begins where the last one ended.

Learns who you are

Code style, naming conventions, preferred libraries, communication tone — captured in .flow/soul.md the first time you mention them. Applies to every project.

Learns your project

Architecture decisions, file conventions, test commands, gotchas, the bug from three weeks ago — stored per-project in memory/projects/<name>/.

Picks the right specialist

62 SKILL.md files auto-activate from natural intent. You ask — the right specialist answers. No @skill syntax, no manual routing.

Stays in your control

Approval gates, plan mode, file snapshots and /undo. Optional autonomy via /goal when you want to walk away.

// 03 Memory & Soul

Two markdown files give the agent a permanent body of knowledge about you and your project. They survive restarts, machine swaps, and CLI upgrades. They are version-controllable. They are yours.

SOUL .flow/soul.md

Engineer-wide preferences

Persistent across every project on this machine. Captures who you are as an engineer.

  • Code style (tabs vs spaces, brace style, line length)
  • Naming conventions (snake_case, camelCase, prefixes)
  • Default test framework, lint config, CI expectations
  • Preferred libraries and architectural patterns
  • Communication tone — formal, terse, with comments or without
  • Domain expertise — embedded, web, data, ops
MEMORY memory/projects/<name>/AGENTS.md

Per-project context

Scoped to one repository. The agent reads it before every turn.

  • Architecture decisions (the "why" behind the structure)
  • Build / test / deploy commands specific to this project
  • Conventions enforced by reviewers
  • Known pitfalls and accepted workarounds
  • External services and how to reach them locally
  • Audit log of stage transitions, goals, and big edits

How the agent actually learns

A background soul-keeper watches every turn for stable preferences and project facts. When it sees one, it appends a line to soul.md or AGENTS.md. The next turn loads that file before anything else — so the agent starts every session smarter than it ended the last.

What gets sent to the model on every turn

What growing-with-you actually looks like

Day 1

You mention you prefer 4-space indents, arrow functions, and vitest. The soul keeper appends three lines to soul.md in the background. You don't have to do anything.

Day 4

You ask for a new module. The agent already follows your style. It also remembers from AGENTS.md that this project uses strict-null-checks and routes through a service layer.

Day 14

A reviewer rejects a PR because of a missing telemetry call. You tell the agent. It writes a new entry in AGENTS.md. From now on it adds the call automatically when generating handlers.

Day 30

You start a new project. The agent already knows you — style, frameworks, tone — from the soul. The project memory starts empty and fills as you work. No re-onboarding.

A real soul.md excerpt — appended automatically by the soul-keeper
## Code style
- 4-space indent, no tabs
- single quotes for strings
- prefer arrow functions; async/await over .then()
- always destructure props in React components

## Tooling defaults
- test runner: vitest (was jest before 2025-11)
- bundler: vite for libs, next for apps
- formatter: prettier with print-width 100

## Communication preferences
- terse summaries, full detail in code comments only when non-obvious
- always show a diff before applying multi-file edits
- never add emoji to commit messages
/soul shows the current contents. /memory shows what's injected into this session. Both files are plain markdown — check them into git, edit them by hand, share them with your team.

Autonomous Goal — Plan → Code → Test → Evaluate, persistent across restarts

Define an objective, kick off the runner, walk away. Every turn the auditor inspects git status, counts new tests, optionally runs your verify command, and decides Continue / Complete / Block. State is checkpointed under memory/projects/<name>/goal.json — quit the REPL, the runner survives.

SHIPPING NOW · P0 / P1

Agile crew & ceremony engine

A detachable "management layer" of AI personas that handles the work surrounding the engineering loop — backlog grooming, sprint ceremonies, acceptance reviews, portfolio governance. Toggleable; V-Model-only mode keeps the existing experience untouched.

  • ◐
    Autonomous Agile Persona Engine. PO, Customer, Scrum Master, PM, and PMO as AI agents (or seats for real humans). Each has a distinct toolset, voice, and gate authority. #99
  • ○
    Product Owner agent. Owns the backlog, authors user stories from raw conversations, gates sprint planning. #105
  • ○
    Customer agent. Persona-driven acceptance — simulates a real user during sprint review, signs off (or doesn't) on each story. #106
  • ○
    Scrum Master agent. Runs ceremonies, detects impediments, posts stand-ups, drives retros. #107
  • ○
    Ceremony engine. Sprint lifecycle, configurable cadence, automatic ceremony triggers from tuiBus events. #111
  • ○
    Human-relay system. Blocking gates, REPL modals, and notification relays for when an AI persona needs a real human's call. #110
  • ○
    Agile toggle. Detach the entire crew with one flag; fall back to plain V-Model mode for solo or strict-process projects. #104

  • ○
    Processes, Policies & Procedures. Definition of Done, compliance gates, coding standards — consulted at every stage transition. #101
  • ○
    Templates library. User stories, sprint boards, retro formats, project charters — pre-filled and consistent across every new project. #102
  • ○
    Lessons-learned & historical metrics. Velocity, forecasting, retrospective knowledge base — a real institutional memory that compounds over sprints. #103
  • ○
    Project Manager agent. Budget tracking, cross-goal dependencies, timeline reports. #108
  • ○
    PMO agent. Portfolio governance, standards enforcement, cross-project alignment dashboard. #109
EXPLORING · P1 / P2

Quality, autonomy & rough edges

Smaller items that show up regularly in our own dogfooding — verified deliverables, smarter long sessions, deeper goal/squad integration, and platform polish.

  • ○
    /qa command — verified deliverables gate. Scope validation + acceptance checks between "the agent says it's done" and "we hand it to the PO". #97
  • ○
    Session resume strategy. When you re-open a long-running session, pick between a token-cheap summary and the full transcript — with a comparison preview before you commit. #114 · #95
  • ○
    /goal V-Model mode. Let an autonomous goal drive the SWE.1 → SWE.6 pipeline across multiple turns instead of a single-shot run. #16
  • ○

// 09 Install

From npm (recommended)

$ npm i -g @agenit/cli
$ agenit --help

$ gemini                    # one-time Google OAuth
$ cd ~/projects/my-project
$ agenit init               # auto-detects profile
$ agenit                    # start the REPL

From source

$ git clone https://github.com/muhammed-eldabea/AgenIT.git
$ cd AgenIT
$ bash install.sh           # Linux / macOS
# Windows:
> powershell -ExecutionPolicy Bypass -File install.ps1

Prerequisites

  • Node.js ≥ 20
  • pnpm ≥ 9 — npm i -g pnpm
  • Python ≥ 3.11 — hardware tools only
  • Gemini CLI ≥ 0.37
  • Google account for OAuth
