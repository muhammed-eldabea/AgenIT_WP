// 00 Not an MVP factory

Most AI coding tools optimise for the first working demo. AgenIT optimises for the second-to-last commit — the one where every line traces to a spec, every change is critiqued, and the audit report renders green. Spec-Driven Development on rails, with a paper trail.

spec-driven

SDD before a single line is written

/speckit run <idea> produces spec.md → plan.md → tasks.md with acceptance criteria. /speckit critique hunts vague verbs, contradictions, untestable claims before implementation. No code lands without a referenced requirement.

// 00a Architecture — how the pieces fit

Three layers, one OAuth login. The TypeScript orchestrator never speaks to the LLM directly — it shells out to a backend subprocess (Gemini CLI by default), which calls a Python sidecar for hardware & traceability work. Memory, hooks, MCP, and the marketplace plug into the same canonical paths regardless of which backend is active.

Layer 1

TypeScript orchestrator

17 pnpm packages — the Ink REPL, V-Model state machine, Mission Squad, Goal loop, MCP client/server, federation, sandbox, web UI, VS Code extension. Loads soul + project memory into every prompt; routes free-text turns through the active backend; enforces budgets and approval gates.

Layer 2

LLM backend subprocess

One of Gemini CLI (OAuth, default), Claude CLI (OAuth via the local claude binary), Anthropic SDK, OpenAI, or Ollama. The orchestrator never holds an API key itself unless the chosen backend needs one.

Layer 3

Python sidecar & hooks

.flow/tools/ Python scripts drive JLink, CAN/LIN, Saleae, traceability, codedigest, marketplace install, and the memory index. .flow/hooks/ Python scripts gate tool calls (citation guard, MISRA, slopsquatting, undo snapshot, shell deny-list). Same hooks run under either CLI.

Outside-world connections

  • Google OAuth via Gemini CLI — no API key required for the default path. Token cached in ~/.gemini/.
  • Anthropic OAuth via Claude CLI — same idea, uses Pro / Max / Team / Enterprise subscription if you already have it.
  • MCP Registry — live search across the public Model Context Protocol registry; /marketplace mcp.
  • Claude Code marketplace — browse and install any Claude Code plugin from GitHub; translated into Gemini-compatible artefacts at install time.
  • GitHub MCP — first-class for issues / PRs / commits / CI when wired into .mcp.json.
  • Hardware — JLink (SEGGER) over USB, SocketCAN / PCAN, Saleae Logic, all through .flow/tools/*.py.
  • Peer instances — the @flow/federation package syncs memory + decisions between AgenIT instances over mTLS for team workflows.

// 02 Objective

Most AI coding tools start every conversation from zero. You re-explain the project, the conventions, the build commands, the bug you fixed last week. AgenIT is built around the opposite premise — the agent learns once and remembers, so each session begins where the last one ended.

Learns who you are

Code style, naming conventions, preferred libraries, communication tone — captured in .flow/soul.md the first time you mention them. Applies to every project.

Learns your project

Architecture decisions, file conventions, test commands, gotchas, the bug from three weeks ago — stored per-project in memory/projects/<name>/.

Picks the right specialist

62 SKILL.md files auto-activate from natural intent. You ask — the right specialist answers. No @skill syntax, no manual routing.

Stays in your control

Approval gates, plan mode, file snapshots and /undo. Optional autonomy via /goal when you want to walk away.

// 03 Memory & Soul

Two markdown files give the agent a permanent body of knowledge about you and your project. They survive restarts, machine swaps, and CLI upgrades. They are version-controllable. They are yours.

SOUL .flow/soul.md

Engineer-wide preferences

Persistent across every project on this machine. Captures who you are as an engineer.

  • Code style (tabs vs spaces, brace style, line length)
  • Naming conventions (snake_case, camelCase, prefixes)
  • Default test framework, lint config, CI expectations
  • Preferred libraries and architectural patterns
  • Communication tone — formal, terse, with comments or without
  • Domain expertise — embedded, web, data, ops
MEMORY memory/projects/<name>/AGENTS.md

Per-project context

Scoped to one repository. The agent reads it before every turn.

  • Architecture decisions (the "why" behind the structure)
  • Build / test / deploy commands specific to this project
  • Conventions enforced by reviewers
  • Known pitfalls and accepted workarounds
  • External services and how to reach them locally
  • Audit log of stage transitions, goals, and big edits

How the agent actually learns

A background soul-keeper watches every turn for stable preferences and project facts. When it sees one, it appends a line to soul.md or AGENTS.md. The next turn loads that file before anything else — so the agent starts every session smarter than it ended the last.

What gets sent to the model on every turn

How the scaffolding agent stays on-rails

Each kind locks the agent into a strict output shape so the install step can drop the body into the canonical path without post-processing:

  • Skill — "Output ONLY a SKILL.md — YAML frontmatter (name, description) + Markdown body."
  • MCP — "Output ONLY valid JSON for .mcp.json — an object with mcpServers mapping to { command, args, env }."
  • Agent — "Output ONLY the AGENTS.md block — ## <name> heading + Role, Specialty, Tools allowed, System prompt."
  • Tool — "Output ONLY a Python script — shebang + docstring + run(payload) -> dict + a main() that reads stdin JSON and writes stdout JSON."

Refine mode (r) prepends the existing body plus your feedback so the agent edits in place instead of starting fresh. Half-written drafts persist across REPL exits in ~/.agenit/harnesses/drafts.json (debounced 300 ms).

  • MCP merge — duplicate mcpServers ids replace silently rather than warning. Pick a unique id if you don't want to overwrite.

How install really works

  1. Discover — /marketplace search <q> queries all four sources (curated · Claude Code · MCP Registry · already-installed) and merges the results.
  2. Resolve — the installer figures out the artefact type (skill, MCP server, plugin) and finds its canonical install path.
  3. Validate — for skills it checks the YAML frontmatter; for MCP entries it dry-runs the launch command; for Python tools it checks the run(payload) entrypoint.
  4. Security gate — the slopsquatting hook blocks suspicious package names; MCP server pinning records exact versions so a future install can't silently swap the binary; skill audit flags skills that override existing ones.
  5. Atomic merge — .mcp.json is updated via temp-file + rename so other servers survive; AGENTS.md de-dups blocks by name; .gemini/skills/ writes are sandboxed by name.
  6. Persist — /marketplace installed shows the current snapshot; /marketplace update refreshes registry caches; /marketplace remove <id> reverses the install with the backup it took.

Air-gapped & pinned installs

For environments without outbound network access, AgenIT runs in --air-gapped mode — only locally-cached registries are consulted, MCP servers are pinned to recorded versions, and the slopsquatting hook treats any unfamiliar package name as a hard block. // 03 Use Cases

Twelve concrete recipes — when to reach for AgenIT, and the exact key-presses to get from prompt to outcome. Each card lists when, steps, what you get, and tips.

01

Onboard a fresh project in 5 minutes

onboarding

When: you've just cloned a repo (or started one) and want AgenIT to know who you are and what the project is before you type your first prompt.

  1. Authenticate Gemini once — gemini in any terminal. Opens a browser for Google OAuth. No API key required.
  2. Drop into the project — cd ~/projects/my-app.
  3. Initialise — agenit init. AgenIT auto-detects a profile (web, embedded, automotive, design, generic) from repo markers and seeds .agenit/, .gemini/skills/, and a starter AGENTS.md.
  4. Pick a backend — the picker shows what's installed (Gemini, Claude CLI, OpenAI, Ollama). Hit Enter to take the default.
  5. Run it — agenit. The REPL boots; the right rail shows your soul + memory loaded.

What you get: a project-specific AGENTS.md, hooks wired into the active backend, and a soul-keeper running in the background to learn your preferences from this session forward.

↗ Tip: agenit init --target both seeds both Gemini and Claude CLI so you can swap mid-session with /backend.

02

Build a new feature with the V-Model pipeline

build

When: you have a feature in your head and you want the full Req → Arch → Code → Test → Critic → Audit treatment with traceability tags.

  1. State the goal in plain English — /run add OAuth login flow with Google.
  2. Watch the stages light up — the left-rail pipeline widget shows Req → Arch → Code → Test → Critic → Audit, one at a time.
  3. Approve gates — between stages, AgenIT pauses for your review. Type y or use /yolo on to auto-approve.
  4. Inspect traceability — /traceability shows every SWR-NNN requirement linked to its code (via @req tags) and tests.
  5. Audit at the end — /audit prints coverage, orphan tests, and (for automotive) the ASPICE checklist.

What you get: requirements.md, architecture.md, edits to your code with @req tags, fresh tests, and a clean audit report — all linked.

↗ Tip: for big features, run /speckit run <task> first to produce spec.md / plan.md / tasks.md, then /speckit export-to-swe1 seeds the pipeline.

build

  5. Hand off to V-Model — /speckit export-to-swe1 seeds requirements.md for /run.

What you get: a contract-shaped Python script in the canonical sidecar path — the same place the built-in JLink / CAN / Saleae helpers live.

↗ Tip: if the tool needs deps, add them to .flow/tools/requirements.txt and re-run the project's venv install.

build

  2. Browse a vendor — /marketplace cc-browse anthropics/claude-code-plugins.

↗ Tip: /marketplace installed lists what's currently active — useful before a clean rebuild.

16

Run safely in a regulated / air-gapped repo

autonomy

When: the repo handles regulated data (medical / automotive safety / defence) and you need zero outbound network plus an auditable hook trail.

  1. Boot air-gapped — agenit --air-gapped. Blocks all outbound except the OAuth-refresh handshake.
  2. Set a hard token ceiling — --token-cap 80000 --per command — the loop halts when exceeded.
  3. Lock down installs — the slopsquatting hook rejects suspicious package names before they touch npm/pip/cargo; the MCP-pinning + skill-audit scanner flags unverified entries.
  4. Inspect the hook trail — every guard decision lands in .flow/events.jsonl hash-chained for tamper detection.
  5. Set compliance mode — in agenit.toml [compliance] mode = "soc2"|"gdpr"|"hipaa" tightens PII redaction and citation enforcement.

What you get: a hermetic runtime with deterministic guardrails, a tamper-evident audit log, and zero unexpected network traffic.

↗ Tip: the slopsquatting / MCP-pinning / air-gapped story is documented under docs/security/.

17

Try a risky refactor in a sandboxed worktree

autonomy

When: you want to ask the agent for something speculative — "rewrite the auth layer in Rust" — without polluting your working tree or main branch.

  1. Spawn an isolated worktree — /worktree feat/rust-auth. AgenIT creates .gemini/worktrees/feat-rust-auth/ and switches you in.
  2. Work normally — every edit, test, commit happens in the worktree. Your main checkout is untouched.
  3. Parallelise with overnight — /overnight --parallel 3 -- gemini "make a rust port" runs three worktree-isolated copies side by side, each with its own commit chain.
  4. Adopt what works — if the experiment lands, git merge the worktree branch back. If not, just delete the worktree.
  5. Undo per-write fallback — /undo list shows the last 10 snapshots; /undo <id> restores one.

What you get: a safe playground where bold ideas can fail without cost; the same helper powers /overnight --parallel for multi-worktree runs.

↗ Tip: /plan flips approval-mode on (tool calls need explicit y/n) when you don't trust the next batch of edits.

18

Call AgenIT from Claude Desktop, Cursor, or another agent

memory

When: another agent (Claude Desktop, Cursor, an LLM gateway) wants to read your soul + memory or fire a squad / goal — AgenIT acts as an MCP server to that caller.

  1. Start the MCP server — agenit-memory-mcp ships as a stdio binary that exposes 14 tools (soul read, project memory read, doc append, global memory query, …).
  2. Wire it into the consumer — add an entry to Claude Desktop / Cursor / your gateway's MCP config (~/.cursor/mcp.json or the equivalent).
  3. Read soul from outside — the calling agent invokes agenit_soul_read to load your engineering preferences.
  4. Query global memory — agenit_global_memory_query_tag({ tag: "rust-style" }) pulls cross-project knowledge.
  5. Append back — agenit_global_memory_write({ file, tag, text }) records what the external agent learned, so future AgenIT sessions inherit it.

What you get: AgenIT becomes a memory + autonomy substrate for your whole AI tool chain — one institutional memory, many consumers.

↗ Tip: hardware tools (JLink / CAN / Saleae) are also exposed over MCP — external agents can drive your bench.

19

Wire alerts & auto-actions with Recipes

autonomy

When: you want "if X happens, do Y" automation — e.g. when a goal tick fails, log it; when a hardware probe fails, escalate.

  1. Open the Recipes widget — right rail / center-pane (in the REPL). Three default recipes ship enabled: log goal failures, notify agent failures, escalate hardware probe failures.
  2. Define a recipe — edit .flow/recipes/*.toml: when = "goal.tick.failed", action = "system-message: ⚠ goal {{goalId}} blocked at turn {{turn}}".
  3. Test-fire — in the Recipes widget, press t — runs the action against a synthetic event.
  4. Toggle live — p enables / disables without editing TOML.
  5. Edit inline — e opens recipes.toml in the in-TUI vim.

What you get: declarative, hot-reloadable automation tied to the tuiBus event taxonomy — no extra orchestration layer needed.

↗ Tip: trigger templates support variables ({{goalId}}, {{turn}}, {{tool}}) so messages stay specific.

20

Migrate a codebase to a new language / framework

autonomy

When: the migration is too big for one chat but too repetitive for hand-coding. Days, not minutes.

  1. Anchor with a spec — /speckit run migrate from javascript to typescript. Produces phased tasks.md.
  2. Tighten it — /speckit critique looks for missing edge cases ("how do we handle .d.ts shadowing?").
  3. Spawn the autonomous loop — FLOW_GOAL_TEST_CMD="pnpm test" then /goal Migrate the codebase from JavaScript to TypeScript + /goal start.
  4. Add a CI safety net — run /diff-walk --base main nightly to keep the critic running over each new chunk.
  5. Land cleanly — when the goal completes, /audit + /pr-metrics give you a write-up of AI vs human deltas.

What you get: a turn-by-turn migration with auditor-decided continuation, hard token budget, and per-turn git commits you can revert at any point.

↗ Tip: for multi-developer migrations, share the .specify/specs/<feature>/ folder via git so everyone runs against the same task list.
